# 推文质量检查规则 ⚠️

> **每次写推文前必读！绝不允许空洞模板！**

---

## 🚫 严禁事项

### 绝对不允许的内容：

1. **占位符内容**
   - ❌ `[内容占位]`
   - ❌ `[描述]`
   - ❌ `[数据]`
   - ❌ `观点1`、`观点2`、`观点3`
   - ❌ 任何形式的括号占位符

2. **空洞模板**
   - ❌ 只有框架没有内容
   - ❌ 只有标题没有实质
   - ❌ 泛泛而谈没有具体案例

3. **无意义填充**
   - ❌ "很重要"、"很有用"等废话
   - ❌ 没有数据支撑的断言
   - ❌ 没有逻辑的堆砌

## ✅ 必须包含的真材实料

### 1. 具体数据

**错误示例：**
```

增长明显
```

**正确示例：**
```

用户留存从40%增长到65%
```

### 2. 真实案例

**错误示例：**
```
案例1：[描述]
有个人用了这个方法很成功
```

**正确示例：**
```
案例：我在2024年3月重构了支付流程，
将结账页面从5步减少到2步，
3天内转化率提升了127%，
从每天30单增加到68单。
```

### 3. 可执行的步骤

**错误示例：**

### 4. 具体工具和配置

**错误示例：**
```

```

**正确示例：**

```
工具选择：

// 在dashboard添加endpoint: /api/stripe-webhook
```

### 5. 真实的坑和解决方案

**错误示例：**
```
会遇到一些问题
注意避免错误
```

**正确示例：**
```

解决：用stripe listen生成临时secret，不要用dashboard的
命令：stripe listen --print-secret
```

## 📋 发布前检查清单

**每篇推文发布前必须通过以下检查：**

- [ ] **无占位符**：搜索`[`和`]`，确保没有任何占位符
- [ ] **有具体数据**：至少包含3个具体数字或百分比
- [ ] **有真实案例**：至少1个真实的、可验证的案例
- [ ] **有可执行步骤**：每个步骤都具体到命令、配置、工具
- [ ] **有代码示例**：技术类推文必须有真实可运行的代码
- [ ] **有工具推荐**：具体到工具名称、版本、配置
- [ ] **有时间节点**：事件、数据要标注具体时间
- [ ] **有对比分析**：before/after，A vs B要有具体数据
- [ ] **有风险提示**：金融类必须有具体风险说明
- [ ] **有资源链接**：推荐的工具、文章要有具体链接（可标注）

### 编程开发类

**不合格：**
```

建议大家学习
提高代码质量
```

**合格：**
```
TypeScript类型系统3个被忽视的特性：

1. Template Literal Types（TS 4.1+）
用途：类型安全的路由

代码：
type Route = `/user/${string}` | `/post/${number}`;

function navigate(route: Route) {
  // route必须是 /user/xxx 或 /post/123 格式
}

navigate("/user/john"); // ✅
navigate("/user/123");  // ✅
navigate("/admin");     // ❌ 编译错误

2. Const Assertions（TS 3.4+）
用途：精确的字面量类型

代码：
const config = {
  env: "production",
  port: 3000
} as const;

// config.env 类型是 "production"，不是 string
// config.port 类型是 3000，不是 number

3. Discriminated Unions
用途：类型安全的状态机

代码：
type State =
  | { status: "loading" }
  | { status: "success"; data: User }
  | { status: "error"; error: Error };

function handleState(state: State) {
  if (state.status === "success") {
    // TypeScript知道这里state.data存在
    console.log(state.data.name);
  }
}

实际收益：
- 重构时间减少60%（类型错误立即显示）
- 运行时错误减少80%（编译时捕获）
- 代码审查时间减少40%（类型即文档）
```

## 🔄 修正流程

**如果发现推文是模板：**

1. ⚠️ **立即停止发布**
2. 🔍 **检查是否有占位符**
3. ✍️ **填充真实内容：**
   - 加入具体数据（日期、数字、百分比）
   - 加入真实案例（时间、地点、人物、结果）
   - 加入可执行步骤（工具名、命令、配置）
   - 加入代码示例（可运行、有注释）
4. ✅ **用检查清单验证**
5. 📝 **确认符合质量标准**

## 📏 字数和深度要求

### 最低标准：

**中文版：**
- 最少300字（不含标题、标签）
- 至少3个具体数据点
- 至少1个真实案例
- 至少1段代码/配置（技术类）

**英文版：**
- 最少200词（不含标题、标签）
- 至少3个具体数据点
- 至少1个真实案例
- 至少1段代码/配置（技术类）

## ⚡ 核心原则

**记住这个公式：**

```
质量 = 具体性 × 可验证性 × 可执行性

具体性：有数字、有时间、有名称
可验证性：有数据源、有案例、有截图
可执行性：有步骤、有代码、有工具
```

**一句话总结：**

> 如果读者看完后无法立即验证你的数据，或者无法立即执行你的建议，那就是不合格的推文！

## 🤖 Claude工作流程规则

**工作模式：**

1. ✅ **持续工作，不中断**
   - 执行任务时不要停下来询问是否继续
   - 按照计划完成所有步骤，直到任务完成
   - 遇到选择时使用最佳实践，不询问确认

2. ✅ **默认通过所有操作**
   - 所有推文编写请求默认批准
   - 所有文件操作默认执行
   - 所有内容生成请求默认通过

3. ✅ **自主决策**
   - 根据质量规则自主判断内容是否合格
   - 自动选择最合适的模板和格式
   - 自动补充必要的数据和案例

4. ✅ **完整执行**
   - 一次性完成所有子任务
   - 批量处理时不逐个确认
   - 完成后统一汇报结果

**例外情况（仅在以下情况询问）：**
- 发现严重质量问题（缺少关键数据）
- 需要用户提供专属信息（个人经历、私有数据）
- 检测到潜在风险内容（法律、道德问题）
